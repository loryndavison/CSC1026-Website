HTTP

  • HyperText Transfer Protocol makes the world wide web work.
    • Clients ask for resources from the servers by assembling and sending a HTTP request message.
    • Servers respond with the appropriate HTTP response message, including any content to be displayed.
  • HTTP requests and responses travel as TCP/IP packets.
  • HTTP is stateless. This means:
    • Each request/response pair is an independent exchange.
    • No protocol level maintenance of state (for scalability)

HTTP Methods

  • The operations get carried out over HTTP
    • HEAD, GET, POST, PUT, DELETE etc.
  • HEAD and GET are the only mandatory operations
  • GET returns the current state and content of the resource
  • HEAD returns the response metadata
Method Use Safe Idempotent Mandatory
HEAD Exchange of request/response headers Yes Yes Yes
GET Request and return of the current state and content of a resource Yes Yes Yes
POST Request entity body to update resource or as input for processing No No No
PUT Server stores entity body contents at request URI location No Yes No
DELETE Deletes identified resource No Yes No

HTTP Headers

  • Headers are the metadata for the request/response exchange
  • Some metadata is generic and applies to both request and response
    • Date - Date/Time stamp for message
    • Cache Control - Instructions for en-route caching
  • Some web applications use custom headers to add comments or annotations to an HTTP message.
  • Be aware that information in headers is not always trustworthy as they can be spoofed.
Request Headers
  • Used by web clients to identify themselves and control how content is returned.
    • Response preferences
    • Additonal request info
    • Conditional headers
    • Constrain server behaviour
Response Headers
  • Used to specify how content is being returned and how it should be handled.
    • Redirection
    • Additonal server info
    • Authentication
    • Caching
